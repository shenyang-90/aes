## Problem Description

### Summary
The `ctrl_busy` signal is incorrectly derived from `status_reg[1]` which is hardcoded to 0, causing the DUAL_RAIL_EN protection logic to be completely bypassed. This allows DUAL_RAIL_EN (CTRL[9]) to be modified during active AES operations, violating the safety requirement.

### Technical Details

**Current Implementation (aes_top.v)**:
```verilog
// Line 93: ctrl_busy is derived from status_reg[1]
wire ctrl_busy = status_reg[1];

// Line 430: status_reg[1] is hardcoded to 0
status_reg[1]  <= 1'b0;  // Reserved

// Lines 477-482: DUAL_RAIL_EN protection logic
if (ctrl_busy) begin  // ❌ Always FALSE!
    // Protection logic never executes
end else {
    ctrl_reg <= pwdata;  // ❌ DUAL_RAIL_EN can be modified anytime!
}
```

**Impact**:
- DUAL_RAIL_EN can be changed while AES operation is in progress
- Violates Design Spec v1.2 Section 8.2.3.3: "DUAL_RAIL_EN can only be changed when STATUS[BUSY]=0"
- Potential safety hazard: Lockstep could be disabled mid-operation

### Expected Behavior

Per Design Spec v1.2:
- BUSY status should reflect actual operation state
- DUAL_RAIL_EN modification should be blocked when BUSY=1
- Safe switching only allowed when idle

---

## Root Cause Analysis

1. **Incorrect BUSY signal source**: Using `status_reg[1]` instead of actual busy state
2. **Hardcoded reserved bit**: `status_reg[1]` is always 0
3. **Missing busy logic**: No proper busy state tracking in RTL

---

## Reproduction Steps

1. Start AES operation (set CTRL[0] START=1)
2. While operation is running (BUSY should be 1)
3. Write to CTRL[9] DUAL_RAIL_EN via APB
4. **Bug**: DUAL_RAIL_EN changes immediately without protection

**Test Case**: tc_safety_fsm_timeout (SM-RECOV-001B)

---

## Proposed Fix

### Option 1: Use core_busy signal (Recommended)
```verilog
// Add to aes_top.v
wire core_busy = /* logic from aes_controller or internal state */;
wire ctrl_busy = core_busy || (state != IDLE);  // Comprehensive busy detection

// DUAL_RAIL_EN protection
if (ctrl_busy) begin
    // Block DUAL_RAIL_EN change
    ctrl_reg[31:10] <= pwdata[31:10];
    ctrl_reg[8:0]   <= pwdata[8:0];
    // Bit 9 unchanged
end else begin
    ctrl_reg <= pwdata;  // Safe to change DUAL_RAIL_EN
end
```

### Option 2: Fix status_reg[0] BUSY logic
```verilog
// Current (wrong)
status_reg[0]  <= ctrl_reg[0] || core_start;  // Both are pulses

// Fixed
status_reg[0]  <= (state != IDLE) && (state != DONE);  // Reflect actual busy state
```
